# 개발자 GMP 문서 작성 가이드

GMP 문서 작성에서 개발자(모바일 앱 담당)의 역할과 담당 문서 정리.

---

## 1. 문서 흐름 이해

### 1.1 역할별 문서 담당

```
┌─────────────────────────────────────────────────────────────────┐
│                    GMP 문서 작성 흐름                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   기획자 작성              개발자 작성              QA 작성      │
│   ───────────────────────────────────────────────────────────   │
│                                                                 │
│      SRS            →        SDF/SDS        →     테스트 기록   │
│   (무엇을 만들지)        (어떻게 구현했는지)      (검증 결과)     │
│                                                                 │
│   • 기능 요구사항          • 아키텍처 설계         • 테스트 케이스│
│   • 비기능 요구사항        • 함수 단위 설계        • 테스트 결과  │
│   • 사용자 시나리오        • SRS-SDF 매핑         • 버그 리포트  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 입력과 출력

| 구분 | 내용 |
|------|------|
| **입력** | SRS (기획자가 작성한 요구사항 명세서) |
| **출력** | SDF, SOUP 목록, CVR, TAR (추적성) |

## 2. 개발자 담당 문서

### 2.1 핵심 문서: SDF (소프트웨어 상세 설계 파일)

**목적**: SRS의 각 요구사항을 "어떻게" 구현했는지 설계 수준에서 기술

**작성 내용**:
- 소프트웨어 아키텍처 설계도
- 화면별 주요 기능 설명
- 함수 단위 설계 기술서 (SRS ID ↔ 함수 매핑)

**작성 형식** (Canmore 기준):

| SRS ID | 주요 기능 | 기능 설명 | SDS-ID | 함수명 | 함수 기능 설명 |
|--------|----------|----------|--------|--------|---------------|
| SRS-P021 | 복약 일정 조회 | 처방된 약물의 복용 일정 조회 | SDS-P030 | MedicationSchedule.render() | 복약 일정 화면 표시 |
| | | | SDS-P031 | useMedicationSchedule() | 일정 데이터 조회 훅 |
| | | | SDS-P032 | formatScheduleData() | 일정 데이터 포맷팅 |

### 2.2 SOUP 목록

**목적**: 외부 라이브러리(Software of Unknown Provenance) 목록 및 위험 분석

**작성 내용**:
- package.json의 dependencies 목록
- 각 라이브러리의 용도, 버전, 라이선스
- 위험 분석 (의료기기 안전성에 미치는 영향)

**예시**:

| SOUP명 | 버전 | 용도 | 라이선스 | 위험 등급 | 비고 |
|--------|------|------|----------|----------|------|
| react-native | 0.73.x | 앱 프레임워크 | MIT | 중 | 핵심 프레임워크 |
| zustand | 4.x | 상태관리 | MIT | 저 | UI 상태만 관리 |
| @tanstack/react-query | 5.x | 서버 상태 | MIT | 중 | API 통신 |

### 2.3 CVR (코드 검증 기록)

**목적**: 코드 품질 검증 활동 기록

**작성 내용**:
- 코드 리뷰 기록 (PR 리뷰)
- 정적 분석 도구 결과 (ESLint, SonarQube 등)
- 발견된 이슈 및 해결 내역

### 2.4 TAR (추적성 분석 기록)

**목적**: 요구사항 → 설계 → 테스트 간 추적성 보장

**작성 내용**:

| SRS ID | SDF ID | 테스트 ID | 상태 |
|--------|--------|----------|------|
| SRS-P021 | SDS-P030, P031, P032 | TC-P021-01, 02 | 완료 |
| SRS-P022 | SDS-P033, P034 | TC-P022-01 | 완료 |

## 3. SDF 작성 실무 가이드

### 3.1 코드에서 정보 추출

React Native 프로젝트 기준:

```
src/features/medication/
├── screens/
│   └── MedicationScheduleScreen.tsx  → SDS-P030 render()
├── hooks/
│   └── useMedicationSchedule.ts      → SDS-P031
├── components/
│   └── ScheduleCard.tsx              → SDS-P033
└── utils/
    └── formatSchedule.ts             → SDS-P032
```

### 3.2 함수 설명 작성 팁

**나쁜 예**: "데이터를 처리한다"
**좋은 예**: "서버에서 받은 복약 일정 데이터를 화면 표시용 포맷으로 변환한다"

### 3.3 SRS-SDF 매핑 규칙

- 하나의 SRS 요구사항 → 여러 SDF 함수 (1:N 관계)
- 모든 SRS 요구사항은 최소 1개 이상의 SDF와 매핑되어야 함
- 누락된 매핑 = 추적성 결함 → 식약처 심사 시 문제

## 4. 작업 프로세스

### 4.1 권장 작업 순서

```
1. SRS 문서 검토
   └─ 기획자가 작성한 요구사항 파악

2. 코드 분석
   └─ 각 요구사항이 어떤 파일/함수로 구현되었는지 파악

3. SDF 작성
   └─ 아키텍처 → 화면별 기능 → 함수 단위 설계

4. SOUP 목록 작성
   └─ package.json 분석 → 위험 분석

5. 추적성 검증
   └─ SRS ↔ SDF 매핑 누락 없는지 확인
```

### 4.2 작업 도구

- **문서 형식**: Google Docs (협업) 또는 Word
- **코드 분석**: IDE + Claude Code
- **추적성 관리**: 엑셀 또는 전용 도구
